# Кейс: Коворкинг — Требования

## Функциональные требования (FR)

| ID | Требование |
|----|------------|
| FR-01 | Система должна отображать план зала с местами |
| FR-02 | Система должна показывать статус места: "свободно", "забронировано", "занято" на выбранную дату |
| FR-03 | Сотрудник может забронировать свободное место на конкретную дату |
| FR-04 | Система должна проверять, что на одно место — одна бронь на дату |
| FR-05 | Горизонт бронирования — не более 2 недель от текущей даты |
| FR-06 | Сотрудник может отменить свою бронь (статус "отменена", а не удаление) |
| FR-07 | Администратор может управлять списком мест (добавлять, редактировать, блокировать) |
| FR-08 | Администратор может смотреть статистику по бронированиям (количество, процент неиспользованных) |
| FR-09 | Система должна автоматически отменять брони, если сотрудник не пришёл (порог 5% неиспользованных) |
| FR-10 | При попытке забронировать уже занятое место система возвращает ошибку 409 Conflict |

## Нефункциональные требования (NFR)

| ID | Требование |
|----|------------|
| NFR-01 | Загрузка плана зала — не более 3 секунд |
| NFR-02 | Система доступна только через веб-браузер (мобильная версия не поддерживается) |
| NFR-03 | Система должна выдерживать до 50 одновременных пользователей |
| NFR-04 | Данные о бронированиях должны храниться не менее 1 года |
| NFR-05 | Время отклика API — не более 500 мс
