# 🔌 Практика: Проектирование и тестирование API (Swagger UI & Postman)

## 📌 Паспорт модуля
* **Цель**: Освоить практическую работу с интерфейсами Swagger UI и Postman для описания интеграционных контрактов взаимодействия микросервисов, проведения ручного тестирования и выполнения авторской приемки фич у разработчиков.
* **Стек технологий**: REST API, HTTP-методы (GET, POST), Query-параметры, JSON-структуры данных, Swagger/OpenAPI, Postman (Collections, Environments).

---

## 🐱 1. Интерактивная витрина Swagger UI (Кейс «Cat as a Service»)

Для понимания концепции «интерактивного меню» методов был спроектирован и протестирован демонстрационный REST API сервис — *Cat as a Service*, возвращающий динамические изображения котов с наложением пользовательского текста.

### Спецификация OpenAPI метода:
* **Метод / HTTP Verb**: `GET`
* **Базовый URL / Endpoint**: `/api/v1/cats/random`
* **Назначение**: Получение случайного изображения кота с возможностью кастомизации параметров отображения.

### Query-параметры запроса (Параметры фильтрации):
1. `type` (string, optional) — Формат обрезки и вывода картинки (`square`, `full`).
2. `filter` (string, optional) — Цветовой фильтр изображения (`mono`, `sepia`, `negative`).
3. `text` (string, required) — Текст, который система должна автоматически сгенерировать и наложить поверх изображения кота.

Через встроенную кнопку **Execute** в Swagger UI аналитиком проводится первичная техническая верификация: формируется реальный HTTP-запрос к серверу и валидируется структура ответа.

## 🧪 2. Тестирование контрактов в Postman

Для ручного тестирования и отладки интеграционных потоков была разработана параметризованная коллекция запросов в Postman.

### Пример POST-запроса: Создание новой карточки животного

* **URL**: `https://vet-lapka.ru`
* **Headers**: `Content-Type: application/json`

#### Тело запроса (`Request Body JSON`):
```json
{
  "name": "Барсик",
  "type": "cat",
  "breed": "British Shorthair",
  "age_months": 24,
  "owner_id": 1054
}
```

#### Валидация ответа сервера:
При успешной обработке транзакции сервер возвращает статус **`200 OK`** (или `201 Created`) и эхо-JSON с присвоенным системным идентификатором:
```json
{
  "pet_id": 89452,
  "status": "active",
  "created_at": "2026-07-01T03:00:00Z"
}
```

### 💡 Лучшие практики использования Postman в системном анализе:
* **Переменные окружения (Environments)**: Использование синтаксиса `{{base_url}}` в строке адреса позволяет мгновенно переключать коллекцию тестов между тестовым контуром (`QA-stage`), локальной машиной разработчика (`localhost`) и боевым сервером (`production`), исключая ручную перезапись путей.
* **Автоматизация приемки**: Описание структуры JSON-ответа в Postman выступает в роли «источника правды» для фронтенд- и бэкенд-разработчиков, позволяя им вести параллельную сборку интерфейса и логики без конфликтов интеграции.
