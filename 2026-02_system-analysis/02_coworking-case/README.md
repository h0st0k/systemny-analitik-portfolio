# 🏢 Кейс: Система бронирования рабочих мест в коворкинге ИТ-хаба

## 📌 Паспорт проекта
* **Бизнес-цель**: Оптимизировать использование пространства ИТ-коворкинга, автоматизировать учет рабочих мест и устранить до 5% потерь ресурсов, возникающих из-за «забытых» и неиспользованных броней.
* **Задача аналитика**: Разработать структуру базы данных (ER-диаграмму), спроектировать статусную модель жизненного цикла бронирования, исключающую физическое удаление данных, и описать логику интеграционного API контракта для обработки ошибок параллельных запросов (`409 Conflict`).

---

## 🗄️ 1. Логическая модель данных (ER-диаграмма)

Система построена на реляционной СУБД (PostgreSQL) со связями в нотации Crow's Foot:

### Сущности и атрибуты:
1. **`User` (Сотрудник)**
   * `id` (INT, PK) — уникальный идентификатор сотрудника.
   * `full_name` (VARCHAR) — ФИО.
   * `email` (VARCHAR, UNIQUE) — корпоративная почта.
   * `role` (VARCHAR) — роль в системе (`Employee`, `Admin`).
2. **`Workspace` (Рабочее место)**
   * `id` (INT, PK) — номер/идентификатор места.
   * `room_id` (INT, FK) — привязка к залу.
   * `type` (VARCHAR) — тип места (`Standard`, `With_Monitor`, `Meeting_Room`).
   * `status` (VARCHAR) — текущее состояние доступности (`Active`, `Maintenance`).
3. **`Booking` (Бронь)**
   * `id` (INT, PK) — уникальный номер бронирования.
   * `user_id` (INT, FK) — автор брони.
   * `workspace_id` (INT, FK) — зарезервированное место.
   * `booking_date` (DATE) — дата бронирования.
   * `status` (VARCHAR) — статус брони (`New`, `Confirmed`, `Cancelled`, `Expired`).
   * `created_at` (TIMESTAMP) — дата/время создания транзакции.

### Связи между таблицами:
* `User` \(\rightarrow\) `Booking`: Связь **«один-ко-многим» (1:M)**. Один сотрудник может создать множество броней на разные даты, но у одной конкретной брони автором является ровно один пользователь.
* `Workspace` \(\rightarrow\) `Booking`: Связь **«один-ко-многим» (1:M)**. Одно рабочее место может бронироваться многократно в разные дни, но в рамках одной записи брони фиксируется ровно одно место.

## 🔄 2. Жизненный цикл и статусная модель

Для сохранения историчности данных и ведения продуктовой аналитики в системе **категорически запрещено безвозвратное физическое удаление строк (`DELETE`)** из таблицы `Booking`. Отмена операции реализована через логическое удаление по статусной модели:


(New)  ──────>  (Confirmed) ──(Время вышло / No-Show)──> (Expired)│                     │└─────(Отмена)────────┴───> (Cancelled) (Слот мгновенно освобождается)
* **Автоотмена (Оптимизация ресурсов)**: Если сотрудник не подтвердил присутствие на рабочем месте в течение первых 30 минут со старта забронированного времени, система автоматически переводит бронь в статус `Expired` и мгновенно выпускает слот на рынок. Это позволило вернуть в оборот до 5% ранее теряемых рабочих мест.

## 🔌 3. Проектирование API контракта и гонка за ресурсом

### Метод создания брони: `POST /api/v1/bookings`

При обработке параллельных запросов от двух сотрудников, одновременно пытающихся занять одно и то же рабочее место на одну дату, система использует механизм оптимистичной блокировки БД и возвращает ошибку конкурентного доступа.

#### Пример тела запроса (`Request Body JSON`):
```json
{
  "user_id": 1054,
  "workspace_id": 42,
  "booking_date": "2026-07-15"
}
```

#### Сценарий Успеха (`HTTP 201 Created`):
```json
{
  "booking_id": 78912,
  "status": "Confirmed",
  "workspace_id": 42,
  "booking_date": "2026-07-15",
  "message": "Рабочее место успешно зарезервировано."
}
```

#### Негативный сценарий — Гонка за ресурс (`HTTP 409 Conflict`):
Возникает, когда слот времени успел перехватить и провести другой поток транзакции БД на микросекунду раньше.
```json
{
  "error_code": "RESOURCE_ALREADY_BOOKED",
  "http_status": 409,
  "message": "Ошибка бронирования. Выбранное рабочее место №42 на дату 2026-07-15 уже было успешно зарезервировано другим пользователем.",
  "timestamp": "2026-07-01T03:00:00Z"
}
```
